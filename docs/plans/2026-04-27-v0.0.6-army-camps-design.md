# v0.0.6 Army, Camps, and Hunger Game Over Design

## Context

**Prompt:** “/start army size as a resource… v0.0.6: army size is a proxy for HP. Start with 5. The only way to lose it is to run out of food. Replenish army size in camps for free (random 1..2). Add game over when army reaches zero. Make stepping on swamps and mountains cost 2 food instead of 1. Use sprite #100 for army. Add camp names + lore pools. Also consider whether signposts should point at camps.”

**Reasoning:** v0.0.5 established a clean pattern for “resources + cooldown PoI + deterministic flavor” (food + farms) while preserving determinism via a pure reducer and explicit `world.*` PoI arrays. v0.0.6 extends that approach by adding an HP-proxy resource (`armySize`), a new cooldown PoI (Camps), a deterministic hunger rule, and a hard fail state (game over) — without pushing the prototype into “full economy/combat”.

---

## Overview

v0.0.6 introduces:

- **Army size (HP proxy):** starts at **5** and is rendered as the primary resource above food (sprite **#100**).
- **Hunger damage:** entering a tile costs food; when you cannot pay the cost, you lose **exactly 1 army** (inspiration-aligned).
- **3 Camps (tile #36):** stepping onto a ready camp yields **+1..2 army** (deterministic) and **+2 food**, then the camp enters a **3-move cooldown**.
- **Terrain pressure:** stepping onto **mountains (#6)** and **swamps (#12)** costs **2 food** to enter (others cost 1).
- **Game over:** when `armySize` reaches **0**, the run enters a game-over state: movement is disabled; Restart still works; a game-over message is shown.
- **Signposts:** signposts point to the nearest Point of Interest (PoI): **Castle**, **named Farm**, or **named Camp** (tie-break: Castle > Farm > Camp).

Non-goals for v0.0.6: gold, towns/shops, combat, paid recruiting, or “opportunity events” for harsh terrain (captured as backlog follow-up).

Behavior note (explicit): game over is evaluated **after** the full MOVE resolution (movement cost + tile on-enter effects). This means a Camp recruit on the destination tile can “save” a move where hunger damage would otherwise drop `armySize` to 0.

- `world.camps: { position: Vec2; name: string }[]` (length **3**, stable order)

- `kind: 'resourceDelta'`

Renderer displays deltas near the corresponding resource value. (If this is too much refactor for a single slice, we can keep `foodDelta` and add `armyDelta` as a second anim kind.)

## World generation (deterministic)

Starting point is `generateWorld(seed)` which currently:

1. Builds base terrain via NOISE quantization into `WALKABLE_COSMETIC_TILE_IDS`
2. Places **1 Castle**
3. Places **3 Farms**
4. Places **SIGNPOST_COUNT** signposts
5. Picks start position

Update to:

1. Build base terrain (remove tile #36 from the cosmetic pool)
2. Place **1 Castle**
3. Place **3 Farms**
4. Place **3 Camps** (tile #36), distinct from castle/farms/signposts and from each other
5. Assign **3 unique camp names** deterministically from the fixed pool (without repeats within a world):
   - Maintain `remainingNames = [...CAMP_NAME_POOL]`
   - For each placed camp (in `world.camps` order):
     - If `remainingNames` is non-empty:
       - `pick = randInt(rngState, remainingNames.length)` (consume RNG)
       - `rngState = pick.rngState`
       - `name = remainingNames.splice(pick.value, 1)[0]`
     - Else: `name = 'A Camp'` (future-proof)
6. Place **SIGNPOST_COUNT** signposts, distinct from castle/farms/camps
7. Pick start position

Known trade-off (accepted for v0.0.6): no “minimum spacing” heuristics yet for camps; placement is purely random-with-retries like other specials so we can observe clustering.

## Reducer rules

### NEW_RUN(seed)

Initialize:

- `resources.food = INITIAL_FOOD` (existing v0.0.5 behavior)
- `resources.armySize = INITIAL_ARMY_SIZE`
- `resources.farmNextReadyStep = Array(world.farms.length).fill(0)`
- `resources.campNextReadyStep = Array(world.camps.length).fill(0)`
- `run.isGameOver = false`

### MOVE(dx, dy)

If `run.isGameOver === true`, ignore MOVE (no state change).

Otherwise on MOVE:

1. Compute destination (wrap), `tileId`, and `stepCount += 1`.
2. Compute food enter-cost:
   - default **1**
   - **2** when destination tile is **mountains (#6)** or **swamp (#12)**
3. Apply food cost and hunger damage:
   - If `food >= cost`: `food -= cost` and emit a **food delta** for `-cost`.
   - Else:
     - `food = 0` and emit a food delta for `-(previousFood)` **only if** `previousFood > 0` (no `-0` delta)
     - `armySize = max(0, armySize - 1)` and emit an **army delta** for `-1`.
4. Apply tile on-enter outcome (farm/camp/signpost/castle/terrain):
   - This stays handler-driven (pure, deterministic) via the existing tile enter registry (`src/core/tiles/*` + `getOnEnterHandler(tileId)`):
     - Default terrain: use `TERRAIN_MESSAGE_BY_TILE_ID` (no RNG)
     - Castle: set found flag + castle message (no RNG)
     - Signpost: message uses nearest-PoI among castle/farms/camps (pure math; no RNG)
   - Farms remain as in v0.0.5 (food gain + cooldown; RNG only on successful harvest).
   - Camps:
     - Identify camp index by `world.camps` position match.
     - Readiness predicate (same semantics as farms): ready iff `stepCount >= campNextReadyStep[i]`.
     - If ready:
       - Gain `gain ∈ [1..2]` deterministically (consume RNG).
       - Pick recruiting line deterministically (consume RNG after gain).
       - Apply `armySize += gain` and `food += CAMP_FOOD_GAIN` (currently **+2**).
       - Set `campNextReadyStep[i] = stepCount + 3`.
       - Message: `"<CampName> Camp\n<RecruitingLine>"`
     - If not ready:
       - Message: `"<CampName> Camp\n<EmptyLine>"`
       - Empty line selection must **not** consume RNG (deterministic index function like farms).
5. End-of-move game over check (after step 3 and step 4):
   - If `armySize === 0` after the tile on-enter effects are applied:
   - Set `run.isGameOver = true`
   - Replace `ui.message` with a deterministic pick from the game-over pool (game-over message wins over the tile’s message).
     - Selection must **not** consume RNG.
     - Deterministic selection rule (exact): let `k = (world.seed + stepCount) | 0`, then pick:
       - `idx = ((k % GAME_OVER_LINES.length) + GAME_OVER_LINES.length) % GAME_OVER_LINES.length`
       - `line = GAME_OVER_LINES[idx]`

Tentatively rejected (document for future balance): “remainder-to-army” hunger damage where unpaid food cost reduces army by the remainder. Chosen rule is “exactly 1” to match inspiration and keep the early tuning gentler.

## Camps: flavor determinism (no RNG on empties)

Match the v0.0.5 farm approach:

- **Recruiting (ready)** consumes RNG (already needed for `gain`):
  - Consume RNG for `gain` first.
  - Consume RNG for recruiting line index second.
- **Empty (cooldown)** does not consume RNG:
  - Use a deterministic index function like:
    - `k = (world.seed + campIndex * 7 + stepCount) | 0`
    - `idx = ((k % lines.length) + lines.length) % lines.length`

---

## Signposts: nearest PoI (Castle, named Farm, named Camp)

When stepped onto, compute nearest PoI from the signpost position.

- Candidates:
  - Castle at `world.castlePosition`
  - Each farm at `world.farms[i].position` (name: `"<FarmName> Farm"`)
  - Each camp at `world.camps[i].position` (name: `"<CampName> Camp"`)
- Distance: torus-shortest Manhattan distance (existing helpers).
- Tie-breaks (deterministic):
  1. Prefer **Castle**
  2. Then **Farm**
  3. Then **Camp**
  4. If multiple of the same kind tie, pick earliest array order (`world.farms` / `world.camps`)

Message format (two-line):

- Line 1: `<PoI Name>` (e.g. `Ember Cross Camp`)
- Line 2: `<DIR>, <D> leagues away.`

## Backlog note (mountains/swamps)

Mountains/swamps can cluster under NOISE gen; with higher food costs they become “danger zones”. In a future slice, harsh terrain should have associated opportunities (bonus events, loot, or higher encounter odds) so it feels like risk/reward rather than punishment.

## Acceptance Criteria

| # | Scenario | Expected result |
|---:|---|---|
| 1 | Start a new run | `armySize` starts at **5**; `isGameOver=false`; camps exist in world |
| 2 | Move onto normal tile | food decreases by **1**; no army loss |
| 3 | Move onto mountain/swamp | food decreases by **2** when possible |
| 4 | Hunger damage | if food is insufficient for entry cost, food becomes **0** and army decreases by **1** |
| 5 | Camp placement | exactly **3** camp tiles exist; they don’t overlap castle/farms/signposts; names are unique within the run |
| 6 | Step onto a ready camp | army increases by **1..2** and food increases by **2**; camp enters cooldown; message uses recruiting pool |
| 7 | Step onto camp in cooldown | no army gain; message uses empty pool; RNG state does not advance |
| 8 | Camp cooldown expires | after **3** moves since recruit, the same camp can pay out again |
| 9 | Game over | when army reaches **0**, `isGameOver=true`, movement is disabled, and a game-over message is shown |
| 10 | Restart after game over | Restart starts a new run even when `isGameOver=true` |
| 11 | Signpost message | points to nearest PoI among castle/farm/camp with specified tie-breaks; two-line format |
| 12 | Determinism | same seed + same click sequence yields identical outcomes (placements, gains, messages) |
