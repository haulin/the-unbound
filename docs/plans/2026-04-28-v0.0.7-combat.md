# v0.0.7 Combat + Contextual Buttons (Implementation Plan)

Plan status: **implemented** on branch; checklist below is archival (do not treat unchecked boxes as open work).

> **For agentic workers:** prefer `/use-subagents` or `/execute` for checkpointed work.

## Context

Combat is the primary future gold source, but for **v0.0.7** we only reward **food** (gold later). Combat should feel like Mannzerhacker’s asymmetric “army vs army” model:

- Percentage-ish hit chances via contested rolls
- Your hit halves the enemy numbers
- Their hit removes one of yours
- **Return/Flee** is always available (cost tuned during v0.0.7 iteration)

We also need **contextual buttons**: the right 3×3 grid becomes context-driven (combat replaces overworld navigation), while preserving existing best practices:

- Single plain state + pure reducer
- Determinism: no `seed(time())`; RNG consumption is controlled and explicit
- OCP: add new entries/functions rather than touching many switches

## Goal

Implement deterministic combat encounters, contextual right-grid buttons, and combat preview rendering:

- **Henge** is a new feature tile (sprite **#66**) that **always** starts combat on enter.
- **Woods/Mountain** tiles can start combat with **~20%** chance on enter.
- Combat rewards **food** (no gold yet).
- Right grid in combat shows **Fight** (sprite **#74**) and **Return** (sprite **#76**) and disables movement.
- Left panel shows a distinct combat preview (enemy sprite **#102**), enemy count, and enemy delta animation.

## Non-goals (v0.0.7)

- Gold resource, shops, towns
- Complex enemy types, equipment, critical hits
- Flee failure / “escape chance” mechanics

## Determinism rules

- **Ambush chance** must not consume `world.rngState` (so it doesn’t perturb farm/camp payouts).
- Starting combat **does** consume `world.rngState` (enemy size roll).
- Each **Fight** action consumes `world.rngState` (two contested rolls).

## Combat rules (concrete)

### Starting combat

When combat starts, spawn enemy size:

- `enemyArmy = playerArmy + randInt(world.rngState, playerArmy + 1).value`
  - Range: `[playerArmy .. playerArmy*2]`

### Fight round

On `FIGHT`:

- Roll `w = randInt(rng, playerArmy + 5)`, then `b = randInt(rng, enemyArmy + 5)`
- If `w.value >= b.value`: **player hits**
  - `nextEnemy = floor(enemyArmy / 2)` (so `1 -> 0`)
  - `killed = enemyArmy - nextEnemy`
- Else: **enemy hits**
  - `armySize -= 1`

Combat reward:

- Paid **once**, when the enemy is eliminated:
  - `food += randInt(rng, COMBAT_REWARD_MAX - COMBAT_REWARD_MIN + 1) + COMBAT_REWARD_MIN`

On `RETURN`:

- Exit combat and lose **1 troop**.
- If that drops `armySize` to 0, the run ends (game over).

On exit (victory or return), show an encounter “exit message” line (deterministic pool) instead of restoring tile flavor.

## Encounter spawning

### Henge (guaranteed)

On `MOVE` enter where destination `cell.kind === 'henge'`: always start combat.

### Woods/Mountain ambush (~20%)

On `MOVE` enter where destination `cell.kind === 'woods' | 'mountain'`: start combat if `shouldStartAmbush` returns true (`src/core/combat.ts`):

- `cellId = pos.y * world.width + pos.x` (via `cellIdForPos`)
- Hash is **`hashSeedStepCell({ seed, stepCount, cellId })`** inside `prng.ts` (same family as farm/camp/signpost-style deterministic picks — **no `world.rngState` consumption** for ambush).
- `(hash % 100) < COMBAT_AMBUSH_PERCENT` (default 20). Exact roll vs older inline formulas may differ; still deterministic from seed + step + cell.

## Contextual buttons (platform-agnostic contract)

Introduce a pure core builder for the right grid:

- `getRightGridCellDef(state,row,col) -> { iconKey?, tilePreview?, action? }`
- `iconKey` is semantic: `goal|minimap|restart|fight|return|enemy` (no sprite IDs in core)
- Platform maps `iconKey -> spriteId` (TIC-80 uses #74/#76/#102 etc)

### Combat right-grid layout (TIC-80)

- `(1,0)`: **Fight** (`iconKey:'fight'` → sprite #74)
- `(1,2)`: **Return** (`iconKey:'return'` → sprite #76)
- `(1,1)`: enemy icon (`iconKey:'enemy'` → sprite #102), non-clickable
- `(0,1)` and `(2,1)`: disabled for now
- Corners (Goal/Minimap/Restart) stay available

## Rendering (TIC-80)

Left panel behavior while in combat:

- If `ui.leftPanel.kind === 'auto'` and `encounter.kind === 'combat'`:
  - Illustration shows the current tile as a full **64×64** preview
  - Overlay an “enemy stats” plate:
    - enemy icon **#102** + enemy count
    - enemy delta anim near enemy count (mirrors food/army delta overlays)

Right-grid mode transition (enter/leave combat):

- The right-grid can play a short transition when switching between overworld ↔ combat.
- Reveal order is **N→W→S→E→C** on both enter and exit (same order in both directions).
- Tentatively rejected: reverse exit order (**C→E→S→W→N**) is cool, but not chosen for v0.0.7.

## Acceptance criteria

- Entering a **Henge** tile always starts combat.
- Entering **woods** or **mountain** starts combat with ~20% chance (deterministic by seed + stepCount + cellId).
- Combat UI replaces navigation, but corners remain available:
  - Fight (#74), Return (#76), Enemy icon (#102)
- Fight win uses `enemy = floor(enemy/2)` (so `1 -> 0`) and rewards **5..15 food** once on victory.
- Return exits combat and costs **1 troop**.
- Enemy spawn is scaled to the player: `[player .. 2*player]`.
