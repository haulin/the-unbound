# v0.0.8 — UI polish (9-slice borders, lore color, grid hover, texture overlay)

Target: TIC-80 cart (`src/platform/tic80/`). Core game logic (`src/core/`) should stay unchanged unless tests need fixture updates.

---

## Acceptance criteria

### 9-slice border

- A **single reusable** routine draws a framed rectangle (corners + edges) given screen bounds and slice metadata (sprite ids, border thickness in pixels).
- **Left HUD**: the full left panel (`PANEL_LEFT_WIDTH` × `SCREEN_HEIGHT`) uses this frame instead of a plain `rect` + `rectb` outline for the outer chrome.
- **Small-box fallback**: if width/height is too small for independent corner tiles (e.g. inner area ≤ 0), fall back to **`rectb` only** or clipped partial draw—**no negative dimensions, no crashes**.
- **Sprite IDs / layout**: 3×3 frame sprites are pre-allocated in the cart at `#146–148`, `#162–164`, `#178–180` (row stride 16). Border thickness is **one sprite** (8px) and edges **tile/repeat** (not stretch).

### Lore text color

- **PoI name** prints in a **light grey** (dedicated constant).
- **PoI description / narrative**: `s.ui.message` body text (wrapped lines) prints in a **grey** (dedicated constant, slightly dimmer than the name) readable on `UI_COLOR_BG`. Both should be easy to tweak later via constants.
- **Message split rule**: the first line of `s.ui.message` (up to the first `\\n`) is treated as the “name/title” and the remainder as the body.
- **Special headlines** (`GAME OVER` / `YOU WIN` from `s.run`) keep their existing semantic colors; only the narrative block uses the lore color.

### Button hover

- While the pointer is over a **right-grid cell** that has an **action** (`getRightGridCellDef(...).action !== null`), draw a **visible tint fill** hover treatment **behind the icon** (does not cover/obscure the tile/icon).
- **Transparency requirement**: hovered-cell sprites must treat **color `0` as transparent** so the tint fill remains visible behind icons/tiles.
- **End-state disable**: in win/lose (`s.run.isGameOver` or `s.run.hasFoundCastle`), directional movement actions are disabled, so they should not hover-highlight.
- **No hover** for cells with `action: null` or empty/disabled semantics (e.g. `(0,2)` corner).

### Preview texture overlay

- The 64×64 preview area is overlaid with sprite `#133` as an **8×8 tiling of an 8×8 texture tile** (scale 1×).
- The overlay uses a configurable transparency key (`TEXTURE_OVERLAY_TRANSPARENT_COLOR`) so the grain can use black safely.
- The overlay applies to: normal tile preview, combat preview, `LEFT_PANEL_KIND_SPRITE`, and game-over tombstone preview.

## Task breakdown

1. **`UI_COLOR_POI_NAME` + `UI_COLOR_POI_DESC` (+ `UI_COLOR_GRID_HOVER_TINT`)** — `uiConstants.ts`; choose initial Sweetie16 indices that read well on `UI_COLOR_BG` (easy to tweak later).
2. **9-slice draw helper** — New module e.g. `src/platform/tic80/nineSlice.ts` (or a `drawNineSliceFrame` section in `render.ts` if tiny): inputs `(x, y, w, h, borderPx, spriteIds)`, branch on `w >= 2*border` and `h >= 2*border`; else `rectb` fallback. Edges tile in 8px steps. **Export** enough for unit tests (see verification).
3. **Wire sprite IDs** — Use the existing 3×3 block: `#146–148`, `#162–164`, `#178–180` (row stride 16). Define constants (or a `baseSpriteId`) near other UI sprite constants so usage is obvious.
4. **Apply 9-slice to left HUD** — In `drawLeftPanel`, replace the outer `rectb(0, 0, …)` chrome with 9-slice (keep `rect` fill for background unless the 9-slice center is meant to fill).
5. **Lore color split** — In `drawLeftPanel`, print PoI name/header with `UI_COLOR_POI_NAME`; print wrapped narrative lines (`s.ui.message`) with `UI_COLOR_POI_DESC`. Keep win/lose headline color logic unchanged.
6. **Hover state (tint fill)** — Determine hovered grid cell from mouse position; for actionable cells only, draw a subtle tinted fill behind the icon (before `spr`). Sample mouse once (via `input.ts`) and pass it into render so hover + click stay consistent. Ensure both static and animated right-grid draw paths get the same hover highlight. Use colorkey `0` for hovered-cell sprites so the tint reads.
7. **Preview texture overlay** — Overlay the 64×64 preview with a tiled 8×8 texture using sprite `#133`, keyed off a configurable transparency color.
8. **Rebuild + regression** — Run `npm run verify`; spot-check in TIC-80: left frame tiling looks correct, lore readability (name lighter than desc), hover tint shows only on actionable cells and doesn’t obscure icons, preview overlay applies in all preview modes.

## Risks / gotchas

- **Sprite asset drift**: if the 3×3 block at `#146–148`, `#162–164`, `#178–180` changes, the frame will render incorrectly (confirm in the cart before calling it done).
- **Hover draw order**: tint must be drawn **behind** the icon and must not fight any masking/clipping used by right panel rendering.
- **Mouse sampling**: avoid sampling mouse state in multiple places (keep one source of truth so hover and click agree).
- **Overlay cost**: tiling an 8×8 texture over 64×64 draws 64 sprites per frame; keep the overlay sprite simple.
