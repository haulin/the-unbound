## Overview (player-visible behavior)

This version replaces “find the Castle” with “open the Gate”.

- **World has 1 Gate** (replaces the Castle as the primary win-condition tile).
- **World has 1 Locksmith**: the fixed PoI **`Locksmith of the Unbound`** (no name pool).
- **Key acquisition**: stepping onto the Locksmith can trade **10 food** for the **Bronze Key** (once per run).
- **Gate interaction**:
  - Without key: locked, shows hint flavor.
  - With key: opens; run is won (headline stays `YOU WIN`).
- **UI feedback**:
  - When key is obtained, left HUD gets a bronze border highlight.
  - The key icon appears at the **top-right of the whole screen** (reserved slot for future status icons).
- **Signposts**:
  - Candidate PoIs include Gate + Locksmith + existing named PoIs (farms/camps/henges).
  - When choosing a target, ignore candidates at distance \(D \le 2\) **if** any candidate exists farther than 2. (If all candidates are too-close, fall back to the closest.)

### Start-of-run edge cases (explicit)

- A new run starts with the **goal narrative** shown (as today).
- If the starting tile is a **signpost**, the signpost clue is **NOT** appended at start (fix current annoyance). It only appears when you step onto a signpost during play.
- Starting on the **locksmith** or **gate** does **not** auto-buy or auto-win; those behaviors occur on stepping onto them via movement.

## Keyholder concept (brainstorm → decision)

### Are “keyholders” a fantasy staple?

“Keyholder” as a literal named trope is less common than **gatekeepers / wardens / keymasters / jailers / locksmiths**, but “someone controls access by controlling a key” is extremely common fantasy logic. What matters for The Unbound’s tone is:
- the figure feels **mythic/liminal** rather than bureaucratic,
- the exchange feels **inevitable** (ritual, necessity, curse, oath) rather than modern commerce,
- and the mechanic remains future-proof for “food → gold” or “trade → fight”.

### Decision: “Locksmith of the Unbound” (chosen)

The “keyholder” is a folk-myth **locksmith** who can **finish** the key, but needs provisions to do it.

- **Why they have the key:** they keep the blank/pattern; the Gate doesn’t yield to random metal.
- **Why they accept food now:** it’s fuel (for the fire and for the person). Later: gold buys fuel/tools/safety.
- **Why they give it up:** keys are made to leave their hands; the locksmith isn’t the one who crosses.

### Keyholder naming + flavor

- **Name**: always the fixed string **`Locksmith of the Unbound`** (no pool).
- **Flavor text variants** (all deterministic picks; **no `world.rngState` consumption** so this can’t perturb other systems):
  - **First visit / purchase** (player can pay): “the forge wakes / the key is finished”.
  - **Already visited** (player already has key): “the work is done / the forge is cold”.
  - **No food** (player can’t pay): “come back with enough / no heat, no key”.

## Architecture / data model decisions (to satisfy audit)

### State

- **Key possession** lives on `state.resources` as a plain boolean:
  - `resources.hasBronzeKey: boolean`
  - Rationale: tile-enter handlers already receive/return `resources`, so the key can be acquired in a single localized handler without threading new state slices through every call site.

- **Win state** replaces castle-specific `run.hasFoundCastle` with a generic win flag:
  - `run.hasWon: boolean`
  - Rationale: “castle found” is no longer the goal. The reducer and UI already treat `hasFoundCastle` as “run ended in victory”; we’ll rename to the domain-neutral concept.

### World cells / sprite policy

- Introduce new feature kinds:
  - `gate` (closed) → sprite `#68`
  - `gateOpen` (opened) → sprite `#70`
  - `locksmith` → sprite `#72`

- **Gate sprite change mechanism (chosen): mutate the world cell kind.**
  - On opening the gate, the destination cell is updated from `{ kind: 'gate' }` to `{ kind: 'gateOpen' }`.
  - Rationale: `getSpriteIdAt(world,x,y)` and the minimap/right-grid previews are derived from `world.cells`. Mutating the cell keeps the “opened gate” visual consistent everywhere without threading `run.hasWon` into sprite selection.

### Tile-enter contract

To keep win logic localized like farms/camps:
- Replace `TileEnterOutcome.hasFoundCastle?: boolean` with `TileEnterOutcome.hasWon?: boolean`.
- The **gate handler** sets `hasWon: true` only when `resources.hasBronzeKey` is true.

## Signposts (exact selection rules)

Current signpost code scans `world.cells` for candidates. v0.0.9 changes:

- **Candidate set**: all cells with kind in `{ gate, gateOpen, locksmith, farm, camp, henge }`.
- **Distance**: torus-shortest Manhattan distance (existing `torusDelta` + `manhattan`).
- **Too-close skip**:
  - Let `SIGNPOST_MIN_TARGET_DISTANCE = 2`.
  - Compute all candidates with their `d`.
  - If any candidate has `d > 2`, then choose the best candidate among only those (`d > 2`).
  - Otherwise (everything is too close), choose the best candidate among all candidates.
- **Tie-break** (after applying the too-close filter):
  1) Smaller `d`
  2) Lower `kindRank`
  3) Lower `id` (cell id, `y * width + x`) for stability

Proposed `kindRank` (win-path first):
- `gate` / `gateOpen`: 0
- `locksmith`: 1
- `farm`: 2
- `camp`: 3
- `henge`: 4

## Acceptance Specs (GWT)

;===============================================================
; World contains a Gate and a Locksmith.
;===============================================================
GIVEN a new run starts.
WHEN the world is generated.
THEN there is exactly 1 Gate in the world.
THEN there is exactly 1 Locksmith in the world.

;===============================================================
; Buying the Bronze Key.
;===============================================================
GIVEN the player does not have the Bronze Key.
AND the player has at least 10 food.
WHEN the player steps onto the Locksmith.
THEN the player now has the Bronze Key.
AND the player has 10 less food.
AND the message shown is Locksmith “first visit” flavor.
AND the key icon is visible in the top-right of the screen.

;===============================================================
; Cannot buy the key without food.
;===============================================================
GIVEN the player does not have the Bronze Key.
AND the player has less than 10 food.
WHEN the player steps onto the Locksmith.
THEN the player still does not have the Bronze Key.
AND the message shown is Locksmith “no food” flavor.

;===============================================================
; Revisiting after purchase.
;===============================================================
GIVEN the player has the Bronze Key.
WHEN the player steps onto the Locksmith.
THEN the message shown is Locksmith “already visited” flavor.

;===============================================================
; Gate is locked without the key.
;===============================================================
GIVEN the player does not have the Bronze Key.
WHEN the player steps onto the Gate.
THEN the gate does not open.
AND the run is not won.
AND the message hints that a key is required.

;===============================================================
; Gate opens with the key.
;===============================================================
GIVEN the player has the Bronze Key.
WHEN the player steps onto the Gate.
THEN the gate opens.
AND the run is won.
AND the win headline remains “YOU WIN”.

;===============================================================
; Bronze HUD outline indicates key acquired.
;===============================================================
GIVEN the player has the Bronze Key.
WHEN a frame is rendered.
THEN the left HUD shows a bronze border highlight.

;===============================================================
; Signposts avoid “too-close” targets when possible.
;===============================================================
GIVEN the player steps onto a Signpost.
WHEN the signpost chooses a target PoI.
THEN it ignores candidate PoIs at distance 2 or less if any candidate exists farther than 2.
AND it points to the chosen target with correct direction and distance.

## Additions during implementation (recorded for history)

These were added after the initial plan to address observed play feel and reduce future edit sites. Code is the source of truth.

- **Gate/Locksmith minimum separation**: Gate and Locksmith are generated with a minimum torus-Manhattan distance (`GATE_LOCKSMITH_MIN_DISTANCE`). See `src/core/world.ts` and `tests/core/world.test.ts`.
- **Worldgen placement refactor**: consolidated special placement through `placeFeature(...)` (explicit RNG threading + shared “terrain-only” placement guard). See `src/core/world.ts`.
- **Signpost refactor**: selection is single-pass with a stable tie-break (distance → kind rank → id) and a “too-close” skip rule. See `src/core/signpost.ts` and `tests/core/signpost.test.ts`.
- **Start/restart reveal animation**: starting a run enqueues a short grid transition from `'blank'` to overworld. See `src/core/reducer.ts` and `src/platform/tic80/rightGridRenderPlan.ts`.
