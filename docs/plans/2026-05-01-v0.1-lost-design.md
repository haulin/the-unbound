# v0.1 — Lost (Design)

## Context

**Prompt:** `/start v0.1 see docs/backlog.md and docs/the-unbound-learnings.md and docs/lore-and-tone.md`

Backlog v0.1 ("The Map & Scout") was scoped down during brainstorm and split:

- **v0.1 — Lost** (this design): the *lost* mechanic, its trigger, and orientation rules.
- **v0.2 — Map & Scout** (deferred to next minor version): scout companion slot, simplified game-map button, `cell.visited` tracking, scout's PoI-reveal line. The remaining roadmap entries (Towns, Gold, Second Gate) shift down by one minor accordingly.

**Reasoning summary:**

- v0.1 originally bundled three structural changes (lost + first teleport trigger + scout slot). Shipping all three together would have introduced both a new state-machine concept (`knowsPosition`) and a new UI widget (game map) at once. Splitting lets the lost mechanic stabilize on its own before adding visited-tracking and scout dependents.
- Lost is triggered by terrain (woods + swamp), per-tile probabilistic, mutually exclusive with combat ambush via a single deterministic roll per move.
- Mountain stays "the ambush terrain" (combat-only, unchanged identity).
- Swamp gets its first mechanic (lost, pure-risk in v0.1). "Swamp opportunity" (food find / shortcut) is captured in deferred backlog.
- Orientation features in v0.1 are `signpost` and `farm` only — fixed/inhabited/named places that can tell you where you are. Gate, locksmith, camp, henge do **not** orient. (Camps and henges are wilderness; gate is uninhabited; locksmith is a single tile and rarely on the player's path.) Towns will join the orient set in v0.3.

---

## Overview (player-visible behavior)

Stepping onto a swamp or woods tile can teleport the player to a far corner of the world. After teleport, the **coordinate readout shows `??`** instead of `A4`/`C7`/etc. The world is otherwise unchanged: the minimap (debug feature) still shows everything, the message panel narrates the dislocation in lore-y terms, and the next move proceeds as normal — possibly even into another swamp/woods that triggers another teleport.

Walking onto a **signpost** or a **farm** tells the player where they are again — the coordinate readout fills back in.

A run **starts unoriented**: the coordinate readout shows `??` from the very first frame. Even if the starting tile is a signpost or farm, the player must move off and back onto an orienting feature to gain coordinates (consistent with the v0.0.9 "inert start" policy).

(Towns will join the orient set in v0.3 — they're explicitly named, inhabited places by definition.)

The combat-ambush rate on woods is **rebalanced** so total event density on woods stays at 25% (15% fight + 10% lost = 25%, same as today's combat-only number). Mountain is unchanged at 25% fight.

## Architecture / data model decisions (to satisfy audit)

### Run state

```ts
type Run = {
  stepCount: number
  hasWon: boolean
  isGameOver: boolean
  knowsPosition: boolean   // NEW. false at run start.
}
```

Rationale: "do I know where I am" is a run-level mood, sibling to `hasWon`/`isGameOver`. Not on `player` because it isn't a player attribute (it's the player's *information state*).

### Single per-move event roll

Replace the scattered ambush logic in `reduceMove` (and the now-vestigial `shouldStartAmbush`) with one deterministic call per move:

```ts
// src/core/tileEvents.ts
export type TileEventSource = 'woods' | 'mountain' | 'swamp' | 'henge'
export type TileEvent =
  | { kind: 'fight'; source: TileEventSource }
  | { kind: 'lost';  source: 'woods' | 'swamp' }

export function rollTileEvent(args: {
  seed: number
  stepCount: number
  cellId: number
  kind: CellKind
  hengeReady: boolean
}): TileEvent | null
```

Logic:

```
if (kind === 'henge' && hengeReady) return { kind: 'fight', source: 'henge' }

const ambushPct = (kind === 'woods')    ? WOODS_AMBUSH_PERCENT
                : (kind === 'mountain') ? MOUNTAIN_AMBUSH_PERCENT
                : 0
const lostPct   = (kind === 'woods')    ? WOODS_LOST_PERCENT
                : (kind === 'swamp')    ? SWAMP_LOST_PERCENT
                : 0

if (ambushPct + lostPct === 0) return null

const p = percentileForCell({ seed, stepCount, cellId })  // 0..99, deterministic
if (p < ambushPct)              return { kind: 'fight', source: kind }
if (p < ambushPct + lostPct)    return { kind: 'lost',  source: kind }
return null
```

`percentileForCell` is the same hash-and-mod-100 shape as the existing `shouldStartAmbush` (deterministic, no `world.rngState` perturbation, replayable).

**Why one roll, not two:**

- Mutual exclusion is structural, not flag-juggled.
- Total event density per terrain is exactly `ambushPct + lostPct`. Tuning is one number per outcome per terrain — no probability trees.
- New event kinds (loot, traps from deferred backlog) extend the outcome union; reducer dispatch grows by one branch each.

**Henge handled here too** so the reducer has exactly **one** event-roll call site. Henge cooldown (`nextReadyStep`) is already on the henge cell; reducer passes `hengeReady` into the function.

### Tile-enter outcome — orientation

`TileEnterOutcome` gains one optional field:

```ts
type TileEnterOutcome = {
  message: string
  // ...existing fields...
  knowsPosition?: boolean   // NEW. true when entering an orienting feature.
}
```

Handlers updated:

- `onEnterSignpost` → outcome includes `knowsPosition: true`
- `onEnterFarm`     → outcome includes `knowsPosition: true`

Reducer applies it as `OR` (orientation is monotonic during a single move; lost-clearing happens here, lost-setting happens later in the event-dispatch step):

```ts
const nextKnowsPosition =
  prevState.run.knowsPosition || !!outcome.knowsPosition
```

Handlers for gate, locksmith, camp, henge are unchanged — they do **not** orient.

### Teleport landing

```ts
// src/core/teleport.ts
export function pickTeleportDestination(args: {
  world: World
  origin: Vec2
  rngState: number
}): { destination: Vec2; rngState: number }
```

Algorithm:

1. Iterate cells; build the set of candidates whose `kind` is in `TERRAIN_KIND_SET` (non-feature) AND whose torus-Manhattan distance from `origin` is `>= TELEPORT_MIN_DISTANCE`.
2. If non-empty: pick uniformly at random (consume one `randInt`).
3. Else (small map / dense features): clamp — find the largest achievable distance `d_max` for any non-feature terrain tile, pick uniformly among tiles at that distance.

Pure function, explicit `rngState` threading per the v0.0.9 worldgen `placeFeature` pattern.

### Reducer dispatch

`reduceMove` after refactor (sketch):

```
1. Compute next position, food/army cost, baseResources.
2. Run tile-enter handler at next position. Apply outcome (resources,
   message, world, hasWon). Compute nextKnowsPosition (OR).
3. Call rollTileEvent once.
4. If event is null: settle state (no encounter / no teleport).
5. If event is { kind: 'fight' }: existing combat-spawn flow
   (unchanged behavior; just relocated through the new function).
6. If event is { kind: 'lost' }:
     - pickTeleportDestination(world, nextPos, world.rngState)
     - player.position := destination
     - run.knowsPosition := false
     - ui.message := pickLostFlavor({ seed, stepCount, cellId: originCellId })
     - enqueue gridTransition (overworld -> overworld) for visual feedback
     - DO NOT run tile-enter at the destination
     - DO NOT roll another event at the destination
```

### Resource normalization

`normalizeResources` is unaffected. `state.run` normalization (or seeding for legacy / hand-built test states): default `knowsPosition: false` when missing. Reducer `processAction(ACTION_NEW_RUN)` always sets it explicitly.

## Lost-state UI

### Coordinate readout

Today: `formatA1(pos)` is rendered at `render.ts:252` next to a position icon.

Change: introduce

```ts
function formatPositionLabel(s: State): string {
  return s.run.knowsPosition ? formatA1(s.player.position) : LOST_COORD_LABEL
}
```

`LOST_COORD_LABEL = '??'` (constant in `constants.ts` so future copy tweaks are localized — per learnings § "Volatile copy belongs in code").

The position icon and minimap are **unchanged** in v0.1. The minimap is a debug feature today; its evolution (a "real game map" with visited features) is v0.2's job.

### Teleport animation

Reuse the existing `gridTransition` anim with `params: { from: 'blank', to: 'overworld' }`. No new anim kind. Same visual vocabulary as the run-start reveal — the right-grid cells materialize one-by-one at the destination, which is the natural visual for "you appear somewhere new". An `'overworld' → 'overworld'` transition would be visually inert because the renderer resolves both phases against the (already updated) player position. Duration: `gridTransitionDurationFrames()` (already a helper in `reducer.ts`).

`MoveSlideAnim` is **not** enqueued for teleport (you don't slide to the destination — you just appear there).

// Renamed from COMBAT_AMBUSH_PERCENT (was a single shared knob for woods+mountain).
export const WOODS_AMBUSH_PERCENT    = 15  // was 25 split: 15% fight + 10% lost = 25% total

`COMBAT_AMBUSH_PERCENT` is removed; existing references migrate to `MOUNTAIN_AMBUSH_PERCENT` / `WOODS_AMBUSH_PERCENT` per terrain. (Sum guard is enforced by a unit test, not a runtime assertion — TIC-80 runtime.)

## Rejected alternatives

- **Per-handler event rolls** — each terrain handler (`onEnterWoods` / `onEnterMountain` / `onEnterSwamp`) rolls its own events. Rejected: today woods/mountain share `onEnterDefaultTerrain`; per-terrain handlers would be a wider refactor with no clear gain. The single `rollTileEvent` function is enough.
- **Lost as a tile-enter handler outcome (`outcome.teleport: Vec2`)** — handlers return their own teleport target. Rejected: teleport is a position-changing event sibling of combat (which already lives in `reduceMove` via the encounter pipeline), not a local tile effect. Keeping `lost` next to `fight` in the event union is more cohesive.
- **Two independent rolls (combat AND lost)** — separate ambush/lost rolls run in sequence. Rejected by user during brainstorm: stacks total event density (~50% on woods if 25% + 25%). Single roll is structurally mutex.

## Acceptance Specs (GWT)

;===============================================================
; Run starts unoriented.
;===============================================================
GIVEN a new run starts.

WHEN the world is generated.

THEN the player does not know their position.

AND the coordinate display shows the lost label.

;===============================================================
; Stepping onto a signpost orients the player.
;===============================================================
GIVEN the player does not know their position.

WHEN the player steps onto a signpost.

THEN the player knows their position.

AND the coordinate display shows the player's coordinates.

;===============================================================
; Stepping onto a farm orients the player.
;===============================================================
GIVEN the player does not know their position.

WHEN the player steps onto a farm.

THEN the player knows their position.

AND the coordinate display shows the player's coordinates.

;===============================================================
; Gate and locksmith do not orient.
;===============================================================
GIVEN the player does not know their position.

WHEN the player steps onto the gate.

THEN the player still does not know their position.

GIVEN the player does not know their position.

WHEN the player steps onto the locksmith.

THEN the player still does not know their position.

;===============================================================
; Camps and henges do not orient.
;===============================================================
GIVEN the player does not know their position.

WHEN the player steps onto a camp or a henge.

THEN the player still does not know their position.

;===============================================================
; Starting tile is inert (no auto-orient).
;===============================================================
GIVEN a new run starts and the starting tile is a signpost or a farm.

WHEN the run begins.

THEN the player does not know their position.

;===============================================================
; Swamp can teleport the player.
;===============================================================
GIVEN the player knows their position.

WHEN the player steps onto a swamp and the swamp pulls them off course.

THEN the player's new position is a non-feature terrain tile.

AND the new position is at least 4 leagues from the swamp.

AND the player no longer knows their position.

AND a lost flavor message is shown.

;===============================================================
; Woods can teleport the player.
;===============================================================
GIVEN the player knows their position.

WHEN the player steps onto woods and the woods turns them around.

THEN the player's new position is a non-feature terrain tile.

AND the new position is at least 4 leagues from the woods.

AND the player no longer knows their position.

AND a lost flavor message is shown.

;===============================================================
; Combat and lost are mutually exclusive on a single move.
;===============================================================
GIVEN the player steps onto woods.

WHEN the move completes.

THEN the player has either entered combat, OR been teleported, OR neither — never both on the same move.

;===============================================================
; Mountain ambushes are unchanged; mountain does not teleport.
;===============================================================
GIVEN the player steps onto a mountain.

WHEN the tile event resolves.

THEN the player's position remains the mountain tile.

AND the only possible event outcomes are a combat ambush or nothing.

;===============================================================
; Swamp does not start combat.
;===============================================================
GIVEN the player steps onto a swamp.

WHEN the tile event resolves.

THEN the only possible event outcomes are a teleport or nothing.

;===============================================================
; Becoming oriented again clears lost.
;===============================================================
GIVEN the player does not know their position after a teleport.

WHEN the player steps onto a signpost or a farm.

THEN the player knows their position.

AND the coordinate display shows coordinates again.

;===============================================================
; Teleport landing is inert.
;===============================================================
GIVEN the player is teleported by a lost roll.

WHEN the player lands on the destination tile.

THEN no tile effects fire on the landing tile (no farm food gain, no key purchase, no signpost orientation, no encounter).

AND no further event roll occurs on the landing tile.

AND no food or army is consumed by the teleport itself.

;===============================================================
; Lost again is possible.
;===============================================================
GIVEN the player has just been teleported.

WHEN the player makes a new move onto a swamp or woods and is again pulled off course.

THEN the player teleports again.

## Behavioral Impact

- **Modified behaviors:**
  - Woods event density redistributed: was 25% combat ambush; now 15% combat ambush + 10% lost (total still 25%). Existing combat-ambush tests on woods may need percent updates.
  - `Run` shape gains `knowsPosition: boolean`. Any reducer test that constructs a `Run` literal needs updating.
  - `formatA1` callsite in `render.ts` becomes `formatPositionLabel(state)`.
- **New behaviors:**
  - Teleport / lost mechanic on swamp + woods.
  - Coordinate label blanking via `LOST_COORD_LABEL`.
  - Orientation gain on signpost/farm tile-enter.
  - Lost flavor message pool.
- **Existing tests requiring updates:**
  - `tests/core/combat.reducer.test.ts` — any tests asserting `COMBAT_AMBUSH_PERCENT`-driven probabilities; rename + update if asserting on woods rate specifically.
  - `tests/core/v0.0.9-key.acceptance.test.ts` — `makeState()` constructs a `Run` literal; needs `knowsPosition: false`.
  - Any test currently using `shouldStartAmbush` directly will need to use `rollTileEvent`.
