# v0.3 — Gold & Towns (Design)

## Context

**Prompt:** `/start v0.3 see docs/backlog.md docs/the-unbound-learnings.md docs/lore-and-tone.md`

v0.3 adds an economy layer and a new point-of-interest interaction surface:

- **Gold** as a persistent run resource.
- **Town** points of interest with a **modal** purchase/hire loop.
- **Scout hiring moves from camps to towns** (Scout remains a one-time run upgrade).
- A small **UI layout pass** (secondary stats top-right, bordered; right-grid shifted down).

**Tone constraints (from `docs/lore-and-tone.md`):** folk-myth register; practical/worn; no heroic proclamations; respect player intelligence; “quiet” presentation.

**Existing acceptance tests detected:**

- `tests/core/v0.0.9-key.acceptance.test.ts`
- `tests/core/v0.1-lost.acceptance.test.ts`
- `tests/core/v0.2-map-scout.acceptance.test.ts`

---

## Overview

### Gold

- Gold is displayed as part of the run’s resource HUD.
- Gold persists for the run. (The gate is currently game end.)
- **Starting gold:** `0`.

### Combat reward (victory only)

When the player defeats the enemy army (combat ends by victory), the player receives:

- **Gold**: a random amount in **5..15**.
- **Food**: a small bonus in **0..2**.

No reward is paid on fleeing combat.

**Determinism note:** combat rewards consume `world.rngState`. We roll **gold first**, then roll the small **food bonus** (so tests can compute expected `rngState` precisely).

**Run-over guard:** rewards are only paid when the run is not already over (`!run.isGameOver && !run.hasWon`), matching existing reducer guardrails.

### Towns (modal PoI)

- Towns are named PoIs.
- Stepping onto a town enters a **Town encounter** (analogous to Camp/Combat control remaps).
- While in a town encounter, travel (MOVE) is blocked until **Leave**.
- Each town exposes **exactly three distinct offer kinds**, fixed per town for the whole run (one of the four kinds is omitted), chosen from:
  - Buy food.
  - Buy troops.
  - Hire Scout.
  - Buy rumors.
- Purchases are repeatable (multiple buys per visit) if the player can pay.
- Stock is infinite for this phase (no depletion/limits).
- Prices vary by town (food cheaper in some towns, troops cheaper in others).

**If the player cannot afford an offer:** the town shows a “not enough gold” message and nothing changes.

**Move cost note:** stepping onto a town is still a normal MOVE — the player pays the usual movement/enter food cost for that step (same pipeline as other features).

**Tile-event note:** entering a town does not trigger terrain tile events (no ambush, no lost teleport). The town interaction is the only result of that step beyond normal move costs and the town’s own on-enter message.

**Orientation note:** entering a town **orients** the player (`run.knowsPosition = true`), same as farms/signposts. Rationale: towns are stable landmarks and reduce “lost forever” frustration while gold economy is still ramping.

### Rumors (v0.3)

In v0.3, “Buy rumors” is **flavor-only**: pay gold, receive a rumor line. No map reveal and no PoI location hinting yet.

**Deterministic rumor line:**

- Rumor text comes from a fixed pool (e.g. `TOWN_RUMOR_LINES` in `src/core/lore.ts`).
- The line is selected via an existing deterministic picker (seed + town id + step), so it does **not** consume `world.rngState`.
- Multiple rumor purchases without advancing `run.stepCount` may yield the same line; that’s acceptable for v0.3.

### Scout acquisition move (camp → town)

- Scout remains a one-time upgrade stored on the player’s resources (unchanged effects).
- Camps no longer offer “Hire Scout”.
- Towns may offer “Hire Scout” as one of the three offers.
- Scout cost is paid in **gold** (town purchase), not food.

## Approach (recommended)

Add a new `TownEncounter` and keep the reducer as an orchestrator that delegates encounter-specific actions to encounter modules (OCP-style, consistent with v0.2 learnings).

Why this shape:

- It matches the existing “modal encounter via `state.encounter` + right-grid remap” pattern (combat, camp).
- It avoids premature “generic PoI menu framework” refactors (explicitly out-of-scope for today).
- It localizes future expansions (stock limits, tavern rumors with map reveal, etc.) behind the town module without scattering switch edits.

## Data model (core)

### Resources

Extend `Resources` with gold:

```ts
resources: {
  food: number
  gold: number // NEW
  armySize: number
  hasBronzeKey: boolean
  hasScout: boolean
}
```

**Defaulting / normalization:** when constructing state from partial `Resources` objects (tests, future save/load), missing `gold` defaults to `0`.

### World cells (town is a named feature)

Add a `TownCell` variant like other named PoIs, with stable per-town offers/prices:

```ts
type TownOfferKind = 'buyFood' | 'buyTroops' | 'hireScout' | 'buyRumors'

type TownCell = {
  kind: 'town'
  id: number
  name: string
  offers: readonly TownOfferKind[] // length 3, fixed per town for the run; maps to North/West/South in that order
  prices: {
    foodGold: number
    troopsGold: number
    scoutGold: number
    rumorGold: number
  }
  bundles: {
    food: number
    troops: number
  }
}
```

Notes:

- Storing offers/prices on the cell makes “fixed per town” structural and avoids having to derive it on each visit.
- Bundles are part of the “stable per town” contract for now (easy to tune later).
- Purchase semantics are direct: **Buy food** pays `prices.foodGold` and grants `bundles.food`; **Buy troops** pays `prices.troopsGold` and grants `bundles.troops`.

### Encounters

Add a `TownEncounter` variant:

```ts
type TownEncounter = {
  kind: 'town'
  sourceKind: 'town'
  sourceCellId: number
  restoreMessage: string
}
```

Town encounter does not increment `run.stepCount` for purchases; MOVE still increments stepCount.

## Worldgen (town placement)

Add towns as named features during `generateWorld`:

- `TOWN_COUNT` (initial default: 2; tune during balance pass).
- `TOWN_NAME_POOL` in `src/core/lore.ts`.
- Placement uses the existing `placeNamedFeature` helper.
- Town offers/prices are generated at placement time using `world.rngState` so they are stable and replayable for the seed.
- Initial tuning targets (balance pass can adjust):
  - `bundles.food = 3`
  - `bundles.troops = 2`
  - `prices.foodGold` in `3..6`
  - `prices.troopsGold` in `5..10`
  - `prices.scoutGold` in `12..20`
  - `prices.rumorGold` in `3..6`

### Offer-set generation (deterministic and simple)

- Town offer kinds are **always three distinct kinds** (one omitted).
- Generation rule:
  - Define a fixed base order: `[buyFood, buyTroops, hireScout, buyRumors]`.
  - **Guarantee Scout availability:** for the first placed town, the omitted kind is chosen from the three non-scout kinds, so at least one town offers `hireScout` for the run.
  - Roll `omitIndex`:
    - For town #0: `omitIndex = randInt(rngState, 3)` over `[buyFood, buyTroops, buyRumors]` (i.e. never omit `hireScout`).
    - For other towns: `omitIndex = randInt(rngState, 4)` over the full base order.
  - `offers = baseOrder` with the omitted entry removed.
  - Store `offers` on the `TownCell` in that order, so UI mapping is stable.

**RNG consumption order (worldgen):** for each town, roll `omitIndex`, then roll each price field in a fixed order (`foodGold`, `troopsGold`, `scoutGold`, `rumorGold`).

### Worldgen helper seam (to keep RNG threading explicit)

Existing `placeNamedFeature` currently allows RNG consumption for choosing positions and names but does not pass `rngState` through `buildCell`. Towns need additional per-town rolls (offers + prices), so v0.3 will add a dedicated helper:

- `placeNamedFeatureRng` with `buildCell: (x, y, name, rngState) => { cell, rngState }`.

Goal: **town structured fields are generated in one deterministic RNG chain**.

## UI / Rendering (TIC-80)

### HUD: gold display

- Add a gold icon + value alongside army and food.
- v0.3 will include a **gold delta animation** mirroring `foodDelta` (non-blocking), so combat rewards and purchases have immediate feedback.

### Layout pass

- Secondary stats (seed/pos/steps) move to the **top-right**, inside a bordered panel.
- Shift the 3×3 right-grid down to make room (visual breathing room; still centered in remaining area).

### Town encounter controls

Use a stable cross layout like camp/combat:

- North / West / South: the three offers, in the order listed in `TownCell.offers`.
- East: Leave (keep “exit” muscle memory aligned with combat/camp leave).
- Center: town icon (no-op).

Map toggle remains available as a meta corner button as in v0.2.

### Town entry/exit animation (right grid)

Entering a town (on MOVE) and leaving a town (via Leave) enqueue a `gridTransition` animation so the right-grid cross reveals the town UI (and then reveals overworld tiles again on exit).

### Town action surface (core)

Town interaction actions are explicit (analogous to camp actions), so the reducer does not need to decode compass slots:

- `TOWN_BUY_FOOD`
- `TOWN_BUY_TROOPS`
- `TOWN_HIRE_SCOUT`
- `TOWN_BUY_RUMOR`
- `TOWN_LEAVE`

`rightGrid` maps North/West/South to one of the four “offer actions” based on the `TownCell.offers` indices, and maps East to `TOWN_LEAVE`.

### Camp layout change (remove Hire Scout)

When in a camp encounter (v0.3):

- West: Search camp (unchanged).
- East: Leave (unchanged).
- North: disabled (no Hire Scout).
- South: disabled (unchanged).
- Center: camp icon (no-op).

Camp preview no longer shows a Scout cost line.

## Sprite ID strategy (for this phase)

- New v0.3 UI needs additional icons/buttons (gold icon; town options; rumor icon).
- Sprite sheet pressure is real; we will **not** “shuffle IDs mid-phase” unless required for a blocker.
- **Plan:** keep all numeric IDs in one place so shuffles are cheap and localized.

**Working rule (to keep sprite changes cheap):**

- Sprite numbers live only in `src/core/spriteIds.ts` as `SPRITES` (single source of truth).
- Gameplay + UI code references `SPRITES.*` directly (no sprite alias exports like `TILE_*` or `*_SPRITE_ID`).
- Right-grid buttons are rendered from `spriteId` returned by `getRightGridCellDef` (no `iconKey` indirection / mapping table).
- Any post-art “shuffle pass” should be localized to `src/core/spriteIds.ts` without touching reducer logic.

## Post-implementation tweaks (record)

- **Map markers**: towns show as `T`; opened gate (`gateOpen`) still shows as `G` (see `src/core/gameMap.ts`).
- **Signposts**: towns are eligible PoIs and are ranked between farms and camps (see `src/core/signpost.ts`).
- **Rumors**: town rumors are built from `BARKEEP_TIPS` (plus a Locksmith hint), deterministically shuffled per town, and cycle on repeated purchase within one town encounter (see `src/core/town.ts`).
- **Town offer plate**: town preview overlays offer prices using stat icons (food/army) plus a rumor illustration sprite (`SPRITES.cosmetics.rumorIllustration`) (see `src/platform/tic80/render.ts`, `src/core/spriteIds.ts`).
- **Top bar**: the right-panel header spans the button grid width, reserves space for key/scout icons, and is drawn after grid ops so it does not disappear during tile slide (see `src/platform/tic80/render.ts`).
- **Sprite ID refactor**: extracted sprite numbers to `src/core/spriteIds.ts` and removed right-grid `iconKey` mapping in favor of `spriteId` (see `src/core/rightGrid.ts`, `src/platform/tic80/rightGridRenderPlan.ts`).
- **Playtest tuning**: increased starting gold to `INITIAL_GOLD=15`, increased combat gold reward to `8..20` and food bonus to `0..4`, and reduced town scout price range to `8..12` (see `src/core/constants.ts`).

## Acceptance Specs (GWT)

;===============================================================
; Combat victory pays gold (and small food).
;===============================================================
GIVEN the player is in combat.

WHEN the player defeats the enemy army.

THEN the player gains between 8 and 20 gold.

AND the player gains between 0 and 4 food.

;===============================================================
; Towns open a modal encounter on entry.
;===============================================================
GIVEN the player is traveling the overworld.

WHEN the player steps onto a town.

THEN the player is shown town options.

AND the player cannot continue traveling until they choose Leave.

;===============================================================
; Leaving a town exits the encounter and restores travel.
;===============================================================
GIVEN the player is in a town encounter.

WHEN the player chooses Leave.

THEN the town encounter ends.

AND the player can travel again.

;===============================================================
; Towns orient the player (lost -> oriented).
;===============================================================
GIVEN the player is lost (their position is unknown).

WHEN the player steps onto a town.

THEN the player becomes oriented (their position is known).

;===============================================================
; Town offers are fixed per town.
;===============================================================
GIVEN a specific town exists in the world.

WHEN the player visits that town multiple times in the same run.

THEN the set of available offers is the same each time.

AND the prices are the same each time.

;===============================================================
; Town purchases are repeatable with infinite stock.
;===============================================================
GIVEN the player is at a town that offers buying food.

WHEN the player buys food and has enough gold.

THEN the player pays gold.

AND the player gains food.

AND the player may buy food again if they still have enough gold.

;===============================================================
; Buying troops works the same way.
;===============================================================
GIVEN the player is at a town that offers buying troops.

WHEN the player buys troops and has enough gold.

THEN the player pays gold.

AND the player gains troops.

;===============================================================
; Scout hire moved to towns (one-time).
;===============================================================
GIVEN the player is at a town that offers hiring a Scout.

WHEN the player hires the Scout and has enough gold.

THEN the player pays gold.

AND the Scout joins the player.

;===============================================================
; Hiring Scout when already owned yields lore and no change.
;===============================================================
GIVEN the player already has a Scout.

WHEN the player attempts to hire a Scout again.

THEN a lore message is shown.

AND nothing else changes.

;===============================================================
; Rumors are flavor-only in v0.3.
;===============================================================
GIVEN the player is at a town that offers buying rumors.

WHEN the player buys a rumor and has enough gold.

THEN the player pays gold.

AND the player is shown a rumor line.

;===============================================================
; Not enough gold yields lore and no change.
;===============================================================
GIVEN the player is at a town.

WHEN the player chooses a town offer that costs more gold than they have.

THEN a “not enough gold” lore message is shown.

AND nothing else changes.

;===============================================================
; Camps no longer offer Hire Scout.
;===============================================================
GIVEN the player is at a camp.

THEN the camp options do not include Hire Scout.

**Spec count:** 12 specs covering 12 behaviors

## Behavioral Impact

- **Modified behaviors:**
  - Combat victory reward changes from “food reward” to “gold reward (+ small food)”.
  - Scout acquisition changes from “camp action” to “town purchase”.
  - Towns (new PoI) are orienting landmarks (like farms/signposts).
- **New behaviors:**
  - Town PoIs with fixed offer sets, repeatable purchases, and per-town prices.
  - Gold as a run resource.
- **Existing tests requiring updates:**
  - `tests/core/combat.reducer.test.ts` (combat victory reward assertions).
  - `tests/core/v0.2-map-scout.acceptance.test.ts` (hire-scout-at-camp scenarios).
  - `tests/core/rightGrid.camp.test.ts` and rendering references (camp no longer shows hire-scout).
  - Any tests constructing `resources` literals may need `gold` (or rely on normalization defaults).

## Risks (known) + mitigations

- **RNG ordering changes break tests** (combat reward + worldgen town rolls).
  - Mitigation: document RNG consumption order (done) and keep rolls localized; update/extend deterministic tests accordingly.
- **Sprite pressure / ID shuffle churn**.
  - Mitigation: implement with temporary IDs, then do one deliberate shuffle after sprites for v0.3 are final.
- **Balance volatility** (town price ranges vs combat gold drops).
  - Mitigation: keep all tuning constants centralized; expect a small post-implementation balance pass with targeted tests kept stable.

## Implementation seams (checklist for planning)

This design expects changes along these existing seams:

- **Core types/constants**: extend `Resources` (add `gold`), extend `FeatureKind`/`Cell` (add `town`), add town actions and UI sprite IDs.
- **Worldgen**: add town placement and per-town `offers/prices` generation.
- **Reducer + encounter routing**: entering a `town` cell creates a `TownEncounter`; MOVE is blocked until Leave; town actions mutate `resources` and potentially `ui.message`.
- **Right grid**: add a town encounter layout mapping offers to North/West/South and Leave to East.
- **Renderer (TIC-80)**: render gold resource line and gold deltas; implement top-right secondary-stats panel; shift right-grid down.
- **Lore**: add town name pool and rumor lines pool; town messages use deterministic pickers where appropriate (no `world.rngState` consumption for flavor).
- **Bundled output**: if `the-unbound.js` is treated as a build artifact committed in-repo, ensure v0.3 core changes are reflected there as part of the phase.
