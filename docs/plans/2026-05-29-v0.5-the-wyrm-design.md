## Resulting design (what the v0.5 player experiences)

Same seven-step shape as `2026-05-26-v0.5-the-wyrm-design.md`:

1. Run starts. `GOAL_NARRATIVE` mentions the bronze gate. (Unchanged.)
2. Player walks onto the Locksmith without the Blood. **No modal opens.** Tile shows inline `LOCKSMITH_NO_BLOOD_LINES` only.
3. Player learns about the wyrm via signposts (`${name} Cave` is a valid signpost target), `BARKEEP_TIPS.wyrm` rumours bought at towns, or stumbling onto a cave tile while exploring.
4. Player travels to a cave (the lair). Modal combat encounter: **Fight / Pay / Flee.**
5. Player either bleeds the wyrm in combat (Fight reduces wyrm health to 0; gain `'blood'`; wyrm survives), or pays gold (-30 gold; gain `'blood'`; encounter closes), or flees (-1 troop; nothing gained).
6. Player returns to the Locksmith. With Blood + gold/food payment, the modal opens normally. Bronze key is forged; the Blood is consumed; `LOCKSMITH_PURCHASE_LINES` (with the existing quench beats) shows.
7. Player travels to the Gate. Uses key. Wins.

## Refinements from the audit pass

Each subsection records a decision that the original brainstorm doc was silent about (or a re-framing where it was misleading). These are the *deltas* â€” read the 2026-05-26 doc for the underlying rationale.

### 1. Inventory + party rework (`Resources` reshape)

**Decision:** generalize the boolean flag-bag into two string-list collections so v0.9's slot system inherits the right shape and v0.5 doesn't add a fifth boolean we'd have to migrate again.

```ts
export const MAX_PARTY_SLOTS = 3

export type Resources = {
  food: number
  gold: number
  armySize: number
  inventory: string[]   // 'bronzeKey' | 'blood' | (future trinkets)
  party: string[]       // 'scout' | 'mule' | (v0.9: + boar/healer/captain/fisherman/magpie); length <= MAX_PARTY_SLOTS
}
```

- `inventory` replaces `hasBronzeKey` (existing) and absorbs the new `'blood'` (v0.5). Uncapped.
- `party` replaces `hasScout` + `hasTameBeast` (renamed `'mule'` to match the existing `MULE_*` lore pools). Capped at `MAX_PARTY_SLOTS = 3`.
- Continuous resources (`food`, `gold`, `armySize`) stay unchanged.

`BuyGain` reshape (called from town/farm/locksmith reducers):

```ts
type BuyGain = { inventoryAdd?: string[]; partyAdd?: string[] }
```

**Migration scope:** ~13 source files and ~25 test files reference the old booleans. Almost entirely mechanical search-and-replace plus a `makeResources({ inventory?, party? })` test helper. No semantic changes; existing acceptance tests must continue to pass after the reshape.

**Rationale (from audit discussion):**

- Adding `hasBlood` would mean migrating 4 booleans in v0.9 anyway, after the slot count grows.
- Encodes a real semantic distinction: `inventory` is uncapped; `party` has a cap of 3 (the slot system's load-bearing constraint).
- Lets v0.9 focus on slot *semantics* (paired refusals, sprite-flash on event) instead of also restructuring `Resources`.

### 2. Combat behavior injection (no new encounter kind)

**Decision:** the wyrm reuses `CombatEncounter`. No `WyrmEncounter` kind. Per-source variation comes from a `CombatVariantConfig` declared on each combat-source mechanic and looked up at runtime via `state.encounter.sourceCellId -> cell.kind -> mechanic.combatVariant`.

```ts
// In src/core/mechanics/defs/combat.ts (new export):
export type CombatVariantConfig = {
  centerSpriteId: number                                 // right-grid (1,1)
  previewPlateLines: (enc: CombatEncounter) => readonly PreviewPlateLine[]
  encounterLines: readonly string[]                      // shown on encounter open
  victoryLines: readonly string[]                        // Fight victory (enemy bled / dead)
  fleeLines: readonly string[]                           // Flee
  payment?: {
    computeCost: (enemyArmySize: number) => number       // pay-button price
    successLines: readonly string[]                      // Pay success
    noFundsLines: readonly string[]                      // Pay attempted, insufficient gold
    onSuccess: (resources: Resources) => Partial<Resources>
  }
  victoryReward: (resources: Resources, rng: StreamRandom) => Partial<Resources>
}

export const STANDARD_COMBAT_VARIANT: CombatVariantConfig = { /* gold + food bonus loot, no Pay button, enemy sprite */ }
```

Wired on `MechanicDef`: a new optional field `combatVariant?: CombatVariantConfig` on every mechanic that opens a combat encounter (henge, terrainHazards, lair).

**Combat reducer/renderer (one-time edits in v0.5, none afterwards):**

- Right-grid: renders the Pay button at (0,1) iff `variant.payment !== undefined`. Renders `variant.centerSpriteId` at (1,1).
- Preview plate: returns `variant.previewPlateLines(enc)` (1 line for standard, 2 lines for wyrm).
- Fight reducer: on victory, applies `variant.victoryReward(resources, rng)` instead of hardcoded gold-roll.
- New Pay reducer (per `ACTION_COMBAT_PAY` action constant): reads `variant.payment`, gates on funds, applies `variant.payment.onSuccess(resources)`, closes encounter, shows `variant.payment.successLines`.

**Why this shape over a `variant: 'standard' | 'wyrm'` discriminator:**

- Open/Closed: adding goblins or brigand-recruit later is a new config object, not an edit to combat reducer.
- Mechanics-live-in-mechanic-modules: each combat-source mechanic owns its variant config. Combat owns dispatch only.
- Function-injection (`computeCost`, `victoryReward`, `onSuccess`) lets the cost curve, loot, and side-effects evolve per variant without touching combat at all.

### 3. Wyrm encounter â€” concrete shape

- **Initial wyrm health:** 30 (carried in `CombatEncounter.enemyArmySize`; the heart sprite re-frames it visually as health, but the math is identical to brigand combat â€” your design call: "halving with one strike is the design of the game").
- **Pay cost:** flat 30 gold for v0.5 (`computeCost: () => 30`). Function shape is in place for v0.7+ to switch to `(hp) => hp` (linear bribe) or `(hp) => hp * hp` (recruit-quadratic, brigands).
- **Loot on victory:** `victoryReward = (resources) => ({ inventory: [...resources.inventory, 'blood'] })`. No gold drop.
- **Loot on Pay:** same â€” `onSuccess` adds `'blood'` to inventory; gold cost is paid via the standard `buy()` helper or equivalent.
- **Flee:** -1 troop (existing combat flee shape, no change).
- **Right-grid layout:**
  | Cell | Sprite | Action |
  |---|---|---|
  | (0,1) top | `SPRITES.buttons.gold` | `WYRM_PAY` (a new combat action constant, gated on `variant.payment !== undefined`) |
  | (1,0) left | `SPRITES.buttons.fight` | `FIGHT` (existing) |
  | (1,1) middle | `SPRITES.cosmetics.wyrmIllustration` (new, ID 166) | `null` (cosmetic) |
  | (1,2) right | `SPRITES.buttons.return` | `RETURN` (existing flee) |
- **Preview plate (2 lines):**
  - Line 1: `SPRITES.cosmetics.heart` (new, ID 196) + remaining wyrm health.
  - Line 2: `SPRITES.stats.gold` + current pay-button price (= `variant.payment.computeCost(enemyArmySize)`).
- **Standard combat preview plate stays 1 line** (`stats.enemy` + count). No regression.

### 4. Worldgen placement order â€” wyrm first

**Decision:** invert the gateâ†”locksmith dependency so the lair places first (mountain tiles are scarce on a 10Ă—10 torus and can be a single cluster).

Order in `MECHANICS`:

1. **wyrm (lair)** â€” places on any mountain tile. No peer dependencies. Uses `placeFeature` with predicate `here.kind === 'mountain'`. Fallback: throws if no mountain tile exists (worldgen guarantees mountains via noise pass).
2. **locksmith** â€” reads `lairPos` via `findCellByKind(cells, 'lair')`. Places on any terrain tile, with `LOCKSMITH_LAIR_MIN_DISTANCE = 4` (torus-Manhattan).
3. **gate** â€” reads `locksmithPos` via `findCellByKind(cells, 'locksmith')`. Places on any terrain tile, with `GATE_LOCKSMITH_MIN_DISTANCE = 7` (existing constant; the constraint stays on the same value, just enforced from the other side).

The peer-aware assertion currently in `placeLocksmith` (`src/core/mechanics/defs/locksmith.ts:99-113`) moves to `placeGate`: gate asserts that locksmith was placed first.

**Gate â†” wyrm distance:** unconstrained. They may land adjacent â€” narratively fine ("the gate is just past the dragon's stones") and avoids over-constraining placement on a tight mountain budget.

### 5. Lair PoI â€” cell kind and signpost

> **Post-implementation revision (2026-05-29):** the `name` field was dropped from `LairCell` and the per-game name pool collapsed to a singleton constant `LAIR_NAME = 'Cave of the Long Wind'` (mirrors `LOCKSMITH_NAME = 'Locksmith of the Unbound'` â€” the existing singleton-PoI naming pattern). Reasoning: the lair is a *singleton* (one per map), so the henge/town `${proper} ${type-noun}` pattern was the wrong precedent â€” Locksmith and Gate are the right ones. A pre-demo lore polish pass may revisit. Shipped state lives in `src/core/lore.ts` (`LAIR_NAME`), `src/core/types.ts` (`LairCell` shape), and `src/core/mechanics/defs/wyrm.ts` (signpost + tile messages all read `LAIR_NAME`). Spec K's acceptance test asserts the singleton name; ranking is locked by two tie-breaker cases in `tests/core/signpost.test.ts`.

- **New `FeatureKind`:** `'lair'`. New cell type:
  ```ts
  export type LairCell = { kind: 'lair'; id: number; name: string; isBled: boolean }
  ```
  `name` from `WYRM_LAIR_NAME_POOL` (renamed; see Â§ Lore pool inventory). `isBled` flips to `true` when the player draws blood (Fight victory or Pay), gating subsequent visits.
- **Sprite:** `SPRITES.tiles.cave: 38` (new). 16Ă—16 with border + padding. The same sprite renders before and after bleed â€” no visual change to the world tile when the wyrm is wounded.
- **Signpost contribution:** `poiSignpost = { rank: 15, name: (cell) => `${(cell as LairCell).name} Cave` }`. Rank 15 was already documented as the intended Lair rank in `src/core/mechanics/types.ts:19`. Display: `${name} Cave` â€” folk-myth narrator says "cave"; the wyrm-inside knowledge comes from barkeep tips.
- **Signpost rank order recap:** Gate=0, Locksmith=10, **Lair=15**, Farm=20, Camp=30, Town=40, Henge=50.
- **Display name elsewhere:** the encounter's restore message uses the same `${name} Cave` prefix on every line.

### 6. Locksmith no-Blood path â€” engine default, not a new pattern

The 2026-05-26 design's option (D) ("Non-modal â€” walking onto the tile shows inline flavor only") was framed as a debut of a new "skip modal if nothing to do" UX pattern. **It isn't.** The engine has shipped this shape since v0.0 â€” see `src/core/mechanics/defs/henge.ts:30-33`, where `onEnterTile` returns `{ message }` (no `encounter`) when the henge is on cooldown.

**Implementation:** in `onEnterLocksmith`, before opening the modal, check `resources.inventory.includes('blood')`. If false, return `{ message: ${LOCKSMITH_NAME}\n${perMoveLine(LOCKSMITH_NO_BLOOD_LINES)} }` â€” no encounter, no enterAnims. The reducer's existing default render path handles the inline-only display. No new pattern, no new helper.

(The "skip modal if nothing to do" idea in `docs/backlog.md` Â§ Ideas is still open for camp/farm/town generalization where the trigger is "not enough money or cooldown" â€” a separate concern from the Locksmith case, which is just `onEnterTile` returning `{ message }`.)

## Acceptance specs (GWT)

These drive `tests/core/v0.5-the-wyrm.acceptance.test.ts`. Each scenario is one observable behaviour from the player's perspective; implementation language is deliberately absent.

**A. Locksmith without the Blood**

- *Given* the player has no Blood in their inventory and the Locksmith hasn't been visited yet,
- *When* the player walks onto the Locksmith tile,
- *Then* no encounter modal opens, the tile shows only an inline line drawn from `LOCKSMITH_NO_BLOOD_LINES`, and the bronze key is **not** added to the inventory.

**B. Locksmith without the Blood, repeated visit**

- *Given* the player has visited the Locksmith without the Blood and walked away,
- *When* the player walks onto the Locksmith tile again,
- *Then* the same inline-only behaviour repeats; no encounter modal opens.

**C. First arrival at a Lair**

- *Given* the player has never bled this Lair (the world-state Lair cell has `isBled === false`),
- *When* the player walks onto the Lair tile,
- *Then* a combat encounter modal opens with the Wyrm-variant right-grid (Fight at left, Pay at top, wyrm illustration at center, Return at right) and the preview plate shows the wyrm's remaining health and the current pay cost.

**D. Bleeding the wyrm via Fight**

- *Given* an open Wyrm encounter,
- *When* the player presses Fight repeatedly until the wyrm's remaining health reaches 0,
- *Then* the encounter closes, the inventory contains `'blood'`, the source Lair cell flips to `isBled: true`, and the message uses a line from `WYRM_VICTORY_LINES`.

**E. Bribing the wyrm with Pay (sufficient gold)**

- *Given* an open Wyrm encounter and the player has at least 30 gold,
- *When* the player presses Pay,
- *Then* gold decreases by 30, the inventory contains `'blood'`, the source Lair cell flips to `isBled: true`, the encounter closes, and the message uses a line from `WYRM_PAYOFF_LINES`.

**F. Bribing the wyrm with Pay (insufficient gold)**

- *Given* an open Wyrm encounter and the player has fewer than 30 gold,
- *When* the player presses Pay,
- *Then* the encounter remains open, gold is unchanged, the inventory does **not** contain `'blood'`, and the message uses a line from `WYRM_NO_GOLD_LINES`.

**G. Fleeing the wyrm**

- *Given* an open Wyrm encounter,
- *When* the player presses Return,
- *Then* the encounter closes, army size decreases by 1, the inventory does **not** contain `'blood'`, the source Lair cell stays `isBled: false`, and the message uses a line from `WYRM_FLEE_LINES`.

**H. Locksmith with the Blood + gold payment**

- *Given* the player has `'blood'` in their inventory and at least `LOCKSMITH_KEY_GOLD_COST` gold,
- *When* the player walks onto the Locksmith and presses the gold-pay button,
- *Then* the bronze key is added to the inventory, the Blood is removed, gold decreases by `LOCKSMITH_KEY_GOLD_COST`, the encounter closes, and the message uses a line from `LOCKSMITH_PURCHASE_LINES`.

> **Post-implementation note (2026-05-29):** the *Then* clause's "encounter closes" is imprecise. Pre-v0.5 (and unchanged in v0.5), the locksmith reducer's pay-success path leaves `state.encounter` open until the player presses LEAVE â€” the same shape as town/farm/camp. The acceptance test asserts the post-pay state, then takes the LEAVE action explicitly, then steps off + back onto the tile to verify the bronzeKey-held arm of `onEnterLocksmith` runs (spec S11). Read this clause as: "the player has finished paying; the message is from `LOCKSMITH_PURCHASE_LINES`; LEAVE closes the modal."

**I. Locksmith with the Blood + food payment**

- Same as H, with food instead of gold and `LOCKSMITH_KEY_FOOD_COST` instead of the gold cost.

**J. Lair after the bleed**

- *Given* the player has bled this Lair (`isBled === true`),
- *When* the player walks onto the Lair tile again,
- *Then* no encounter modal opens, the tile shows only an inline line from `WYRM_BLED_LINES`, and the inventory is unchanged.

**K. Signpost points to the Lair**

- *Given* a worldgen run that placed at least one signpost within range of the Lair and no closer Locksmith / Gate,
- *When* the player walks onto that signpost,
- *Then* the signpost message includes the Lair's display name `${name} Cave` (drawn from `WYRM_LAIR_NAME_POOL`).
