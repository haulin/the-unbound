---

**Goal:** Ship the v0.5 Wyrm milestone — a new Lair PoI in the mountains, a combat encounter with Fight / Pay / Flee that yields the Blood, and a Locksmith that requires the Blood + gold/food to forge the bronze key. Inventory reshapes to a string-list shape that v0.9's slot system inherits.

**Use case:** A player walks the world, discovers a Lair via signpost or barkeep tip, bleeds the Wyrm in combat (or pays 30 gold to bribe), takes the Blood to the Locksmith, forges the key, and uses it at the Gate. The new shape replaces the prior "find Locksmith → find Gate" two-tile loop with a real middle act.

**Non-goals:**
- No combat-balance rework (combo breaker, reward scaling, streak fix) — deferred to v0.6.
- No donkey-overlaps-steps-stat UI intermezzo — deferred to v0.6.
- No new sprite-flash animation primitive — that's a v0.9 slot-system requirement.
- No Pay button on standard combat — wyrm only for v0.5; brigand recruit lands in v0.7.
- No new lore decision-tree (`first-visit` vs subsequent-visit nuances) on the Lair tile — single `WYRM_ENCOUNTER_LINES` opener until bled, single `WYRM_BLED_LINES` after.
- No mountain-cluster placement constraint — any mountain tile is acceptable.
- No tuning of `WYRM_INITIAL_HEALTH` / `WYRM_PAY_GOLD_COST` past 30 / 30 — first-pass values; tuning is a v0.6+ balance concern.

**Architecture:** Inventory + party become string lists (replacing 3 booleans + adding `'blood'`) so v0.9's slot system inherits the right shape. The Combat encounter type stays single-shape; per-source variation comes from a `CombatVariantConfig` object on each combat-source mechanic — Open/Closed for goblins, brigand recruit, etc. The Lair is a new `'lair'` `FeatureKind`; the wyrm mechanic owns its placement (any mountain tile, with locksmith placed ≥4 from it), encounter wiring, and `isBled` gating on the cell.

**Tech Stack:** TypeScript, Vitest, TIC-80 platform render layer.

**Complexity Budget (soft defaults):**
- ≤ 3 changed source files per task (excluding tests)
- No new runtime dependencies
- No new abstraction (base class, interface, factory, registry, adapter, service layer) with fewer than 2 current concrete callers
- No new architectural layer, persistence, cache, queue, background job, or feature flag

`CombatVariantConfig` busts the abstraction-cap-with-fewer-than-2-callers rule on first land (one current caller — wyrm). **Why simpler won't work:** v0.5 explicitly debuts the Pay-button pattern that v0.7 (brigand recruit) and a future goblin variant will reuse; threading the variation through case-of-source switches inside combat would require touching combat in every later milestone. Open/Closed cost is paid once; the second concrete caller (brigand recruit) lands in v0.7.

**TDD during implementation:** enforce

**ATDD during implementation:** waive — GWT scenarios in the design serve as a coverage tool only; the v0.5 acceptance test file lands as a normal Vitest test, no two-stream RED/GREEN ceremony

- **Combat-reducer refactor (Tasks 9–11) touches a high-traffic file.** Mitigation: existing combat tests stay green throughout (refactor preserves behavior); new tests added in each step.

- **Inventory rename (Tasks 4–5) is a wide refactor.** Mitigation: type-driven — `npx tsc --noEmit` finds every site; existing tests cover semantics. Task 4 is intentionally larger than the soft 2–5 minute target because mechanical search-and-replace doesn't decompose cleanly; the typecheck-then-test cadence keeps it bounded.
- **`'blood'` naming as a string item.** v0.9 will introduce typed-item identifiers; for now strings are fine and the migration path is straightforward (search-and-replace).
- **`CombatVariantConfig` is a new abstraction with one current caller (wyrm).** The complexity-budget bust is justified in the header (brigand recruit lands in v0.7); flagging it again here so any reviewer revisits the decision before merge.
