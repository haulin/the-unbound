## Overview

v0.6 reshapes combat from one-flavor-fits-all into three named variants with distinct difficulty, reward, and recruit behavior, and gives henges persistent identity instead of re-rolled spawns. Four threads:

1. **Variant split** — Brigand (mountains), Goblin (woods), Henge (the stones), Wyrm (lair) each declare a `CombatVariantConfig` carrying round bonuses, spawn formula, reward formula, payment block, and lore pools. The shared combat reducer reads these via the existing `cell.kind → mechanic.combatVariant` lookup. Round math is no longer global.
2. **Enemy registers** — Brigands stay punishing (`+5/+5` round math, gold-skewed). Goblins are softer (`+6/+3`, food-skewed, not recruitable). Henge bands use brigand math with persistent state and a +10g reward bonus.
3. **Recruit affordance** — Reuses the existing wyrm `payment` slot. Always rendered. Brigand/Henge accept recruit when the band is wounded and small (≤5); other states return distinct teaching lines. Goblin always returns "they are not yours to lead". No new modal or grid mode.
4. **Carrying capacity** — Food cap clamps gains only. Existing food sticks through cap drops from army loss. Removes the double-punishment of starving on top of losing troops.

Architecturally the variant config is the backbone; the other threads plug into it (rewards, recruit, henge) or live alongside it (food cap). Tuning numbers are sim-anchored (`scripts/combat-sim.mjs`) and recorded in the constants table at the bottom — they are *evidence-backed but tunable* during playtest.

## Combat variant config (architectural change)

Today's `CombatVariantConfig` declares center sprite, plate lines, line pools, optional payment, and `victoryReward`. Round math (`resolveFightRound`) and enemy spawn live at `startCombatEncounter` callsites — fields that should belong to the variant don't. v0.6 closes the gap.

```ts
type CombatVariantConfig = {
  centerSpriteId: number
  previewPlateLines: (state: State) => readonly CombatVariantPlateLine[]
  encounterLines: readonly string[]
  victoryLines: readonly string[]
  fleeLines: readonly string[]

  // NEW: round-math floors. Default brigand 5/5; goblin 6/3.
  playerRollBonus: number
  enemyRollBonus: number

  // EXTENDED: payment is no longer optional. Eligibility is a pure kind;
  // lines and onSuccess live as siblings, keeping logic and copy decoupled.
  payment: {
    computeCost: (encounter: CombatEncounter) => number
    isEligible: (encounter: CombatEncounter, resources: Resources)
      => 'ok' | 'noFunds' | 'notWounded' | 'tooMany' | 'unrecruitable'
    successLines: readonly string[]
    // Variants only need to populate the kinds their eligibility predicate
    // actually returns. Wyrm only carries 'noFunds'; goblin only carries
    // 'unrecruitable'. Combat reducer falls back to the encounter's existing
    // message if a kind is unexpectedly returned without lines (defensive).
    failLines: Partial<{
      noFunds: readonly string[]
      notWounded: readonly string[]
      tooMany: readonly string[]
      unrecruitable: readonly string[]
    }>
    onSuccess: (resources: Resources, encounter: CombatEncounter) => Resources
  }

  victoryReward: (resources: Resources, rngState: number, encounter: CombatEncounter)
    => { resources: Resources; rngState: number }
}
```

Eligibility returns a string discriminator. The combat reducer dispatches: `ok` runs the success path, every other kind picks from the matching `failLines` pool and leaves the encounter open. Lines stay declarative copy; predicates stay logic.

```ts
type CombatEncounter = {
  kind: 'combat'
  enemyArmySize: number      // current
  initialSpawn: number       // NEW: snapshot at this encounter's open
  sourceCellId: number
  restoreMessage: string
}
```

`initialSpawn` is set at `startCombatEncounter` time and used by recruit-eligibility's "wounded" check (`enemyArmySize < initialSpawn`). For henge re-entry against a wounded persistent group, `initialSpawn` is reset to the wounded count at re-open — recruit must mean "you've struck them this engagement", not "you halved them once five steps ago". The cell stores only `currentGroup`; the encounter snapshots `initialSpawn`.

`resolveFightRound` reads `playerRollBonus` and `enemyRollBonus` from the variant the reducer already fetches per fight action — no new branching, two extra constructor args.

### Per-kind variant routing on `MechanicDef`

Today `MechanicDef.combatVariant` is a single `CombatVariantConfig` shared across all `kinds` the mechanic claims. That worked when every kind a mechanic owned shared one variant (terrain hazards used `STANDARD_COMBAT_VARIANT` for woods, swamp, and mountain alike). v0.6 routes woods to goblin and mountain to brigand on the same `terrainHazardsMechanic`, so the field shape changes:

```ts
type MechanicDef = {
  // ...existing fields...
  combatVariantByKind?: Partial<Record<CellKind, CombatVariantConfig>>
}
```

Single-kind mechanics declare a one-entry map (`{ lair: wyrmCombatVariant }`, `{ henge: hengeCombatVariant }`). Multi-kind mechanics declare per-kind entries (`{ woods: goblinCombatVariant, mountain: brigandCombatVariant }`). `swamp` has no ambush policy and so has no entry — its absence from the map is the ground truth that swamp doesn't open combat.

`MECHANIC_INDEX.combatVariantByKind` flattens the per-mechanic maps into the existing `cell.kind → variant` lookup that combat already uses. The lookup site (`combatVariantForEncounter`) does not change. The duplicate-kind validation in `mechanics/index.ts` continues to enforce that any given `cell.kind` is claimed by exactly one mechanic — and now exactly one mechanic provides its variant.

## The four variants

| Field | Brigand | Goblin | Henge | Wyrm |
|---|---|---|---|---|
| Source mechanics | terrain (mountain) | terrain (woods) | henge | wyrm (lair) |
| centerSpriteId | enemies.enemy (#132) | enemies.goblin (#130 NEW) | enemies.enemy (#132) | centers.wyrm |
| playerRoll / enemyRoll | 5 / 5 | **6 / 3** | 5 / 5 | 5 / 5 |
| Spawn shape (at call site) | rolled `[max(2,p-2)..2p]` | rolled `[max(2,p-2)..2p]` | fresh-roll `[10..40]` (persistence flow in henge def) | `fixedEnemySpawn(WYRM_INITIAL_HEALTH)` |
| Reward gold | `round(spawn + U[-3..3])` | `U[0..2]` | `round(spawn + U[-3..3]) + 10` | grants `'blood'` token |
| Reward food | `U[0..4]` | `round(0.4·spawn + U[-1..1])` | `round(0.2·spawn + U[-1..1])` | none |
| Recruit eligibility | brigand-recruit predicate (see below) | always `unrecruitable` | brigand-recruit predicate | wyrm-pay predicate (`gold ≥ cost ? ok : noFunds`) |
| computeCost | `enemyArmySize²` | `() => 0` (unreachable) | `enemyArmySize²` | `() => WYRM_PAY_GOLD_COST` (encounter ignored) |
| onSuccess | `armySize += enemyArmySize` | `(r) => r` (unreachable) | `armySize += enemyArmySize` | grants `'blood'` token (existing) |
| Lore pool family | `BRIGAND_*` | `GOBLIN_*` | `HENGE_*` | `WYRM_*` (existing) |

The **brigand-recruit predicate** is shared by Brigand and Henge variants:

```
if (gold < cost)                          → 'noFunds'
if (enemyArmySize == initialSpawn)        → 'notWounded'
if (enemyArmySize > RECRUIT_MAX_REMAINING) → 'tooMany'
otherwise                                  → 'ok'
```

Cost = `enemyArmySize²`, gain = `enemyArmySize` troops. Eligible only at remaining ≤ 5 (since `tooMany` fires at 6+), so cost stays in `1..25` gold. Recruit reframes the late game of a fight: do one more risky round, or convert the survivors at a steep but bounded gold price.

**Recruit grants partial loot.** On recruit success, the variant's normal `victoryReward` is drawn and gold/food gains are scaled by the *killed ratio* of the band:

```
scale = (initialSpawn - enemyArmySize) / initialSpawn
gold_recruit = floor(victoryReward.gold × scale)
food_recruit = floor(victoryReward.food × scale)
```

Lore frame: the player claims the loot of the fallen; the survivors join carrying nothing. Without partial loot, the N² cost makes recruit dominantly worse than fighting one more round (you've already paid the troop loss in the wear-down rounds; pay 25g for nothing extra and players just flee or finish). Without the N² cost gate, recruit dominantly *replaces* fighting (cheap troops + full loot via subsequent kills). Together — quadratic cost + scaled loot — recruit becomes a real cost-vs-troop-gain decision.

Resulting decision curve at brigand `initialSpawn = 14`:

| N (wounded) | Cost (N²) | Loot scale | Tactical read |
| --- | --- | --- | --- |
| 1 | 1g  | 93% | "scoop the straggler" — small, profitable |
| 2 | 4g  | 86% | break-even-ish |
| 3 | 9g  | 79% | net cost, small troop gain |
| 4 | 16g | 71% | clear cost, real troop gain |
| 5 | 25g | 64% | expensive, max troop conversion |

The N² cost curve creates a "let them bleed to 1-2 then recruit" incentive for cheap troop conversion. The scaled loot ensures very-low-N recruit isn't free (you still pay something net, except sometimes at N=1 where you scoop nearly-full loot — accepted as the "tip the last one" lore beat).

**Henge.** Same N² cost, same scale; bands span 10..40 so the killed ratio at N=5 ranges 50%-87.5%. Mathematically identical to brigand recruit, wider variance — and because henge gold carries the flat +10 bonus on top of `N + U[-3..3]`, recruit at large initial bands and low N can net positive gold (e.g. `initialSpawn=20, N=4`: cost 16g, scaled loot ~21..26g, net +5..+10g). That's design-consistent with the henge identity ("their road, their toll" — the bands carry generous purses) and reinforces the "let them bleed, then recruit" decision when the band is large.

**Goblin / Wyrm.** Partial-loot scaling does not apply. Goblin remains `unrecruitable`. Wyrm has a fixed-cost payment block and grants `'blood'`, not gold/food.

Diegetic legibility for partial loot is carried by recruit-success lore lines (e.g. "bound coin from the fallen — the living follow you now"), not by the plate.

The **wyrm variant** is the migration's compatibility test. Existing acceptance suite (`tests/core/v0.5-the-wyrm.acceptance.test.ts`) must continue to pass without modification — validating that the new shape is back-compatible for the simplest payment case.

Wyrm's existing `payment` block has `computeCost: () => WYRM_PAY_GOLD_COST` and `onSuccess: (resources) => ...`. Both signatures change in v0.6: `computeCost(encounter)` and `onSuccess(resources, encounter)`. The wyrm implementation just ignores the new `encounter` parameter — the wyrm cost is fixed regardless of `enemyArmySize`, and the `'blood'` grant is unconditional on success. The eligibility predicate becomes `(encounter, resources) => resources.gold >= WYRM_PAY_GOLD_COST ? 'ok' : 'noFunds'`. No behavioral change.

## Henge persistence

`HengeCell` gains `currentGroup: number | null` — the active band's enemy count, surviving across flee, draining under fight, set null after defeat. `nextReadyStep` (existing) keeps its name; the semantic shifts so it gates only the *next fresh band's arrival* after a defeat.

State transitions on player entry:

| Cell state | stepCount vs nextReadyStep | Outcome |
|---|---|---|
| `currentGroup == null` | `≥ nextReadyStep` | **Fresh roll**: roll `U[10..40]`, store as `currentGroup`, open combat with `initialSpawn = roll`, line from `HENGE_ARRIVAL_LINES`. |
| `currentGroup == null` | `< nextReadyStep` | **Empty / defeat-cooldown**: no combat, line from `HENGE_EMPTY_LINES`. |
| `currentGroup != null` | (any) | **Wounded re-entry**: open combat with `enemyArmySize = currentGroup`, `initialSpawn = currentGroup`, line from `HENGE_ARRIVAL_LINES`. |

Combat-side outcomes:

| Outcome | Cell after | Cooldown? |
|---|---|---|
| Flee (Return) | `currentGroup = enemyArmySize` (sync the wounded count) | **No** — natural cost (1 troop + 2 food per re-entry) is sufficient friction. |
| Defeat | `currentGroup = null`, `nextReadyStep = stepCount + HENGE_COOLDOWN_MOVES` | **Yes** — gates next fresh band. |
| Recruit success | `currentGroup = null`, `nextReadyStep = stepCount + HENGE_COOLDOWN_MOVES` | **Yes** — band absorbed, henge empty. |

`HENGE_COOLDOWN_MOVES = 3` is unchanged. Worldgen initial state: `currentGroup = null, nextReadyStep = 0`. First entry rolls the first band; world stream RNG advances exactly when it always did, so the determinism golden needs only a controlled re-snapshot once for the new field on the cell shape.

## Recruit & combat preview plate

The combat right-grid top slot is **always rendered** during combat — no conditional show/hide. Reuses `ACTION_COMBAT_PAY` and `SPRITES.inventory.gold` (consistent with wyrm pay; players have learned "top-slot + gold sprite = pay-with-gold"). Click semantics:

| Eligibility | Reducer behaviour |
|---|---|
| `ok` | Deduct cost, run `payment.onSuccess`, close encounter, animate gold delta + grid transition (existing wyrm path). |
| `noFunds` / `notWounded` / `tooMany` / `unrecruitable` | Encounter stays open; UI message picks from the matching `failLines` pool; no resource change. |

The plate is minimal — TIC-80's plate region fits about three rows of three characters, which doesn't accommodate full reward bands without clipping. v0.6's combat plate carries enemy count always plus a conditional recruit-cost row:

```
[variant enemy sprite] N
[gold sprite]          -<cost>     ← only when payment.isEligible(...) === 'ok'
```

- **Always rendered:** the variant's `centerSpriteId` + current `enemyArmySize`.
- **Conditional second row:** when recruit eligibility is `ok`, a row showing the **gold sprite + `-${cost}`** — same shape as the existing wyrm pay plate. The cost is gold even though the *outcome* is troops; consistency with wyrm wins over disambiguation, and the action-button slot already carries the troop affordance.
- **No reward bands.** Reward shape is learned through play, not the plate. The "information, not discount" pillar takes a hit on first-run discoverability — accepted given the column constraint.

**Wyrm plate.** Already follows this shape (`heart-sprite + N`, `gold-sprite + cost`). Recruitable variants align with it. Standard variant has no preview rows. **Goblin plate.** Goblin is unrecruitable, so the plate is just `[goblin sprite] N` — minimal but consistent.

## Spawn-formula widening

Ambush spawns (woods + mountain) widen the floor:

```ts
function rolledEnemySpawn(playerArmy: number): EnemySpawn {
  return (rngState) => {
    const lo = Math.max(2, playerArmy - 2)
    const hi = 2 * playerArmy
    const span = hi - lo + 1
    const r = RNG.createStreamRandom(rngState)
    return { rngState: r.rngState, enemyArmy: lo + r.intExclusive(span) }
  }
}
```

Concrete bands: `p=3 → [2..6]`, `p=5 → [3..10]`, `p=10 → [8..20]`, `p=20 → [18..40]`. Adds variance on the easy side — occasional `5 vs 3` freebies, never below 2. RNG consumption is unchanged (one int from the stream). Henge spawns are independent, controlled by the henge variant's own `spawnEnemy` (`U[10..40]`).

## Food carry cap rule

```ts
// Removed
export function resourcesWithClampedFoodIfNeeded(res: Resources): Resources

// Added
export function applyFoodCapOnGain(prev: Resources, next: Resources): Resources {
  if (next.food <= prev.food) return next               // shrinks pass through
  const cap = foodCarryCap(next)
  if (next.food <= cap) return next                     // gains under cap pass
  return { ...next, food: Math.max(prev.food, cap) }    // gains over cap clamp; over-cap prev (post-army-shrink) sticks
}
```

Five callsites migrate (`combat.ts`, `reducer.ts`, `farm.ts`, `town.ts`, `camp.ts`). Each captures `prevRes` before mutation and calls `applyFoodCapOnGain(prevRes, nextRes)` for the persisted resources. The cap formula (`2 * armySize`, +50 with mule) is unchanged.

This change is the single fix for "double punishment on combat losses" — and is the only v0.6 thread that touches non-combat reducers.

## Open implementation note

Henge needs the cell-state mutation hook to fire on **three outcomes** that today's `onCombatResolved` doesn't all cover: defeat (clear `currentGroup`, set cooldown — already wired), flee (sync `currentGroup` to wounded count, no cooldown), and recruit-success (clear `currentGroup`, set cooldown — same as defeat). Two wirings:

- **(a)** Add `onCombatFled` and `onCombatRecruited` peer hooks on `MechanicDef` (three siblings; one per outcome).
- **(b)** Broaden the existing hook to `onCombatClosed(world, sourceCellId, finalEnemyArmy, outcome: 'victory' | 'flee' | 'recruit')` — single hook with a discriminator.

**Recommendation: (b)**. Three peer hooks for one set of state transitions is more surface than warranted; a single hook with an outcome discriminator scales cleaner. `/plan` makes the call but this is the expected shape.

## Acceptance Specs (GWT)

```
;===============================================================
; Goblin variant in woods opens distinct combat.
;===============================================================
GIVEN the player has 5 troops, 10 food, and 5 gold.
GIVEN the destination tile is woods and rolls an ambush.
WHEN the player steps onto the woods tile.
THEN combat opens with a goblin band of 3..10 enemies.
THEN the encounter lore reads from the goblin encounter pool.
THEN the player's roll feels softer and the enemy's roll feels weaker than against brigands.

;===============================================================
; Brigand variant in mountains uses the harder round math.
;===============================================================
GIVEN the player has 7 troops on grass with mountain north.
GIVEN the destination mountain rolls an ambush.
WHEN the player steps onto the mountain tile.
THEN combat opens with a brigand band of 5..14 enemies.
THEN the encounter lore reads from the brigand encounter pool.
THEN the player's roll uses the standard floor and the enemy's roll uses the standard floor.

;===============================================================
; Henge holds a persistent group on first encounter.
;===============================================================
GIVEN the player approaches a henge that has never been engaged.
WHEN the player enters the henge tile.
THEN combat opens with a band of 10..40 enemies, rolled and stored on the henge.
THEN the encounter line says a band has come to the stones.

;===============================================================
; Fleeing a henge fight leaves the band wounded with no cooldown.
;===============================================================
GIVEN the player has fought a henge band down from 25 to 12 enemies.
GIVEN the player chooses to leave.
WHEN the player steps off and back onto the henge tile next step.
THEN combat opens with the same wounded band of 12 enemies.
THEN no fresh group has been spawned.

;===============================================================
; Defeating a henge band starts a cooldown for a new group.
;===============================================================
GIVEN the player defeats the henge band.
WHEN the player returns to the henge before the cooldown elapses.
```
